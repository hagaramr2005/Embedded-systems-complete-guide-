Embedded Systems Basics
الأساس اللي كل حاجة بتبدأ منه

🤔 يعني إيه Embedded System أصلاً؟

تخيلي إنك بتحطي كمبيوتر صغير جوّا جهاز عشان يعمل وظيفة واحدة محددة — مش أي وظيفة، وظيفة معينة جداً. ده هو الـ Embedded System.

كلمة Embedded معناها "مضمّن" أو "مدفون جوّا". فالنظام ده مش واضح للمستخدم — إنت مش شايفه، لكنه شغّال جوّا الجهاز بيتحكم فيه.

📌 التعريف الأكاديمي

An Embedded System is a dedicated computer system designed to perform one or a few specific functions, often with real-time constraints, embedded within a larger mechanical or electrical system.

أمثلة من حياتك اليومية:

الميكروويف عندك في البيت — فيه كمبيوتر صغير جوّاه بيتحكم في درجة الحرارة والوقت. التلفزيون الريموت — فيه chip بتترجم الإشارات. المكيف — بيحس بالحرارة ويتحكم في المحرك. الغسالة — بتتحكم في المراحل والوقت والماء.

أمثلة صناعية: روبوتات المصانع، أنظمة الطيران، السيارات (نظام الفرامل ABS، الـ Airbag). طبية: أجهزة ضبط القلب (Pacemaker)، ماكينات الأشعة.

🏗️ مكونات Embedded System الأساسية

Input → Processing → Output — دي المعادلة البسيطة لأي نظام.

Input: Sensors (حرارة، ضغط، ضوء، صوت)، Buttons، Keyboard، Camera.

Processing: Microcontroller أو Processor — ده المخ اللي بياخد القرار.

Output: Actuators (موتور، صمام، ضوء)، Display، Speaker.

Memory: Flash (الكود)، RAM (البيانات المؤقتة)، EEPROM (الإعدادات).

Power Supply: مصدر الطاقة — بطارية أو كهرباء.

Communication: UART, SPI, I2C, Wi-Fi, Bluetooth.

❓ أسئلة شائعة عن تعريف Embedded Systems

هل Embedded System دايماً صغير؟ لأ! ممكن يكون كبير جداً زي نظام التحكم في محطة كهرباء. الحجم مش المعيار — المعيار هو إن وظيفته محددة.

هل لازم يكون Real-Time؟ لأ، مش كل embedded system هو real-time. لكن كتير منهم بيكونوا real-time لأن طبيعة عملهم بتطلب استجابة في وقت محدد.

هل لازم يكون فيه OS؟ لأ! ممكن يشتغل بدون OS — ده اللي بنسميه Bare-Metal Programming. بتكتب الكود مباشرة على الـ hardware.

🔄 System Life Cycle — دورة حياة النظام

من المحاضرات: النظام بيمر بـ 5 مراحل:

  1. Birth (الميلاد): الفكرة والمتطلبات — إيه اللي النظام محتاج يعمله؟
  2. Design (التصميم): تصميم الـ hardware والـ software
  3. Growth (النمو): التطوير والاختبار
  4. Maturity (النضج): الإنتاج والاستخدام الفعلي
  5. Decline (الانحدار): الجهاز بيتقادم وبيتستبدل بأحدث

❌ "Embedded Systems دايماً صغيرة" — غلط! الحجم مش المعيار.

Sensors & Actuators
العيون والأيدي للنظام المدمج

👁️ Sensor يعني إيه؟

Sensor هو جهاز بيحوّل كمية فيزيائية (حرارة، ضوء، ضغط، صوت...) إلى إشارة كهربائية يقدر النظام يفهمها. يعني هو INPUT للنظام.

فكّري فيه كـ"عين" أو "حاسّة" للنظام — بيحسّ بالبيئة المحيطة.

📊 Analog vs Digital Sensors

Analog Sensor

بيدي إشارة متواصلة (continuous) — زي جهد بيتغير من 0 إلى 5V.

مثال: Potentiometer, Temperature sensor (LM35), Microphone

محتاج ADC (Analog to Digital Converter) عشان المعالج يقدر يقرأه.

Digital Sensor

بيدي إشارة رقمية (HIGH أو LOW) — 0 أو 1 بس.

مثال: Push Button, PIR Motion Sensor, Digital Temperature Sensor (DHT11)

مش محتاج ADC — المعالج بيقرأه مباشرة.

🔋 Active vs Passive Sensors

Active Sensor: محتاج مصدر طاقة خارجي عشان يشتغل. مثال: Ultrasonic sensor (بيبعت موجات صوت ويسمع الصدى).

Passive Sensor: بيشتغل بالطاقة الموجودة في الظاهرة اللي بيحسها من غير مصدر طاقة إضافي. مثال: Thermocouple (بيوّلد فولت من فرق الحرارة مباشرة).

⚠️ الخطأ الأكتر شيوعاً في الامتحانات

❌ بعض الطلبة بيخلطوا بين Sensor و Actuator.

✅ القاعدة البسيطة: Sensor = Input (بياخد من البيئة) — Actuator = Output (بيأثر على البيئة)

⚠️ السؤال الرخم: "Potentiometer هو sensor ولا actuator؟" — هو Sensor لأنه بيحول وضع المقبض (physical) لفولت (electrical). لكن في سيناريو تاني ممكن يستخدم للتحكم (variable resistor) — المهم اللي بتكلموا عنه في السياق.

Real-Time Systems
مش معناها "سريع" — معناها "في الوقت المحدد"

⏱️ Real-Time يعني إيه فعلاً؟

أكبر غلطة ممكن تعمليها إنك تقولي "Real-Time = سريع". ده غلط 100%.

Real-Time يعني إن النظام لازم ينهي مهمته قبل أو في وقت معين اسمه Deadline. مش المهم تكون سريع — المهم تكون predictable ومحدد.

💡 مثال توضيحي

نظام Airbag في السيارة — لو حصل حادثة، الـ airbag لازم يتفتح في 30 millisecond بالظبط. لو اتفتح في 100ms — متأخر ومش نافع. لو اتفتح في 20ms — برضو ممكن يكون مشكلة. الـ Timing المحدد هو الأساس.

مفاهيم أساسية:

Deadline: الوقت اللي لازم المهمة تخلص فيه. لو المهمة خلصت بعده — اعتبرها فشلت.

Determinism: إن النظام يتصرف بطريقة متوقعة ومحددة في كل مرة — مش عشوائي. أنت عارف إن المهمة دي هتاخد من X لـ Y time بالظبط.

🔴🟡 Hard Real-Time vs Soft Real-Time

الخاصية Hard Real-Time Soft Real-Time
الـ Deadline صارم جداً — لازم يتحقق 100% مهم لكن لو فات مش كارثة
لو فاتت الـ Deadline؟ فشل كارثي — ممكن يسبب ضرر أو وفاة Performance بتقل بس النظام بيكمل
مستوى الخطورة High — Critical Systems Lower — مقبول بعض التأخير
أمثلة Airbag، Pacemaker، Flight control، ABS Brakes Video streaming، Online gaming، Email system
الـ RTOS المستخدم Hard Real-Time RTOS (guarantees deadlines) Soft Real-Time RTOS أو حتى General OS
⚠️ TRAP من الامتحان الفعلي (سؤال 42-45)

السؤال: "Which best describes a hard real-time system?" — الإجابة: Missing a deadline may lead to catastrophic consequences.

السؤال: "In a soft real-time system, what happens if a deadline is missed?" — الإجابة: The performance degrades, but the system continues functioning.

❌ "Hard Real-Time is more flexible" — غلط، العكس هو الصح.

🖥️ RTOS — Real-Time Operating System

RTOS هو نوع خاص من الـ Operating Systems مصمم خصيصاً للـ Real-Time Systems.

الـ RTOS بيضمن إن المهام بتتنفذ في الوقت الصح — بيعمل Scheduling للـ tasks بطريقة محددة.

أمثلة على RTOS: FreeRTOS, VxWorks, QNX, μC/OS-II.

Hard Real-Time RTOS بيضمن إن المهام الحرجة هتخلص في وقتها — ده اللي بيُستخدم في Airbag و Flight Control.

الـ RTOS مش ضروري في كل embedded system — بعضها Bare-Metal (من غير OS خالص).

🎯 سيناريوهات للتفسير

سيناريو 1: نظام monitoring لدرجة حرارة في مصنع — بيقيس كل ثانية ويعمل alert لو اتعدى حد معين. ده Soft Real-Time لأن لو تأخرت القراءة ثانية مش كارثة.

سيناريو 2: نظام تحكم في سرعة محرك طائرة — لازم يستجيب في microseconds. ده Hard Real-Time لأن أي تأخير يعني كارثة.

سيناريو 3: Audio streaming — لو الصوت بفّ للحظة الـ Quality بتقل بس الأغنية بتكمل. ده Soft Real-Time.

Memory Types
الذاكرة بكل أنواعها — RAM وROM وكل حاجة بينهم

🗂️ Volatile vs Non-Volatile — المفهوم الأساسي

Volatile Memory: بتمسح لما الكهرباء بتقفل. زي الـ RAM — بيحفظ البيانات طول ما فيه طاقة بس. مثال: RAM في الكمبيوتر، لو قطّعت الضوء البيانات اللي في الـ RAM راحت.

Non-Volatile Memory: بتفضل شايلة البيانات حتى لو قطّعت الكهرباء. زي الـ Flash، EEPROM، ROM. مثال: الكود في الـ Arduino محفوظ في Flash — حتى لو فصلت الكهرباء الكود لسه موجود.

النوع Volatile؟ قابل للكتابة؟ كيف بتمسحه؟ السرعة الاستخدام في Embedded

DRAM Volatile نعم تلقائي + يحتاج Refresh أبطأ من SRAM Main RAM في الكمبيوتر
SDRAM Volatile نعم تلقائي + Refresh أسرع من DRAM عادي RAM للأنظمة المعقدة

🧠 ليه SRAM أسرع من DRAM؟

SRAM (Static RAM) بيستخدم Flip-Flops عشان يخزن البيانات. الـ Flip-Flop بيفضل شايل البيانات طول ما فيه كهرباء من غير ما يحتاج Refresh. أسرع وأوثوقية أعلى.

DRAM (Dynamic RAM) بيستخدم Capacitors عشان يخزن البيانات. الـ Capacitor بيفضل بيتفرّغ تدريجياً، فمحتاج Refresh كل حوالي 64ms عشان يحتفظ بالبيانات. ده بيأخر العملية.

لكن DRAM أرخص وأكبر كثافة (density) — يعني في نفس المساحة تقدر تحط ذاكرة أكبر بكتير. عشان كده بيتستخدم للـ Main RAM في الكمبيوتر.

💡 Flash vs EEPROM — الفرق الدقيق

Flash هو في الأصل نوع متطور من الـ EEPROM — بس الفرق في طريقة المسح:

EEPROM بيمسح Byte by Byte — بطيء جداً لكن دقيق.

Flash بيمسح Block by Block (Sector) — أسرع بكتير.

ليه Flash بتستخدم للـ Firmware؟ لأن الكود بيتكتب مرة واحدة كبيرة، ومحتاجين سرعة في القراءة.

ليه EEPROM بتستخدم للإعدادات؟ لأن الإعدادات بتتغير byte by byte (مثلاً تغيير إعداد واحد مش محتاج تمسح كل حاجة).

CPU & Processor Components
مخ النظام — ALU والـ Control Unit والـ Registers

🧠 CPU يعني إيه؟

CPU (Central Processing Unit) هو مخ الكمبيوتر أو النظام. بيشتغل على تنفيذ التعليمات.

Processor في الغالب بيعني نفس الـ CPU — بس في سياق Embedded Systems، الـ Microcontroller بيكون فيه CPU + Memory + Peripherals كلهم على نفس الـ Chip.

من الامتحان: "The brain of any computer system is... CPU (Central Processing Unit)".

🏗️ مكونات الـ CPU

المكوّناسمه الكاملوظيفته

Control Unit — بيتحكم في تسلسل التعليمات — بيجيب التعليمة، يفككها، ويوجه الـ ALU
Registers — مخازن صغيرة جداً جوا الـ CPU — أسرع حاجة في النظام
PC Program Counter بيحفظ عنوان التعليمة الجاية اللي هتتنفذ

Status Register Flags Register بيحفظ نتيجة العمليات (Zero, Carry, Overflow)
📌 من الامتحان

"Which of the following are the two main components of the CPU?" — الإجابة: Control Unit and ALU.

"The control unit of a microprocessor..." — لأ بتخزن data ولأ بتعمل arithmetic — بيتحكم في تسلسل العمليات.

🚩 Flags — الأعلام في الـ Status Register

Zero Flag (Z): بيتشعل (=1) لو نتيجة العملية كانت صفر. مثال: 5-5 = 0 → Zero Flag = 1.

Carry Flag (C): بيتشعل لو فيه Carry (حاجة زيادة اتولدت من عملية الجمع) أو Borrow (في الطرح). مثال: 255+1 في 8-bit → Carry = 1.

Overflow Flag (V): بيتشعل لو النتيجة اتعدت المدى المسموح به للأرقام الموقعة (Signed numbers). مثال: أكبر رقم موجب + 1 → Overflow.

🔄 Fetch / Decode / Execute Cycle

كل تعليمة بتمر بـ 3 مراحل:

1. Fetch: الـ Control Unit بياخد التعليمة من الذاكرة بناءً على الـ PC، وبيحفظها في الـ Instruction Register.

2. Decode: الـ Control Unit بيفكك التعليمة عشان يفهم هتعمل إيه (جمع؟ طرح؟ jump؟).

3. Execute: الـ ALU بينفذ التعليمة والنتيجة بتتحفظ في Register أو Memory.

بعدها الـ PC بيتزود (+1 أو حسب الـ instruction) وبنرجع لـ Fetch تاني.

🔢 Integer Processing vs Floating Point Processing

Integer Processing: الـ ALU العادي بيتعامل مع أرقام صحيحة (integers). سريع جداً.

Floating Point Processing: الأرقام الكسرية (زي 3.14) محتاجة معالجة أصعب. في معالجات كتير فيه وحدة خاصة اسمها FPU (Floating Point Unit) أو Floating Point ALU.

الـ FPU بتسرّع العمليات الرياضية الكسرية — مهمة جداً في تطبيقات Signal Processing، رسومات، AI.

Performance Criteria
MIPS و FLOPS — إزاي بنقيس أداء المعالج

📏 MIPS — Millions of Instructions Per Second

MIPS بيقيس عدد التعليمات اللي المعالج بينفذها في الثانية، بالمليون.

مثال: معالج بـ 100 MIPS يعني بينفذ 100,000,000 تعليمة في الثانية.

هل MIPS دايماً مقياس عادل؟ لأ! لأن مش كل التعليمات متساوية. تعليمة جمع أسرع بكتير من تعليمة قسمة. ممكن معالج يعمل MIPS عالي لكن أداؤه الفعلي على تطبيق معين أقل من معالج تاني أبطأ في MIPS.

MIPS مناسب لقياس Integer Processing.

🔢 FLOPS — Floating Point Operations Per Second

FLOPS بيقيس عدد العمليات الكسرية (Floating Point) اللي المعالج بيعملها في الثانية.

GFLOPS = مليار، TFLOPS = تريليون FLOPS.

FLOPS بيستخدم لقياس أداء Supercomputers، Graphics Cards، AI processors، DSP systems.

لو التطبيق بتاعك فيه حسابات رياضية كتير (زي معالجة الصور أو الصوت) → استخدم FLOPS.

DMA & JTAG
نقل البيانات بذكاء + أداة الـ Debugging

🚀 DMA — Direct Memory Access

DMA (Direct Memory Access) هو جهاز خاص بيسمح للـ Peripherals إنها تنقل بيانات من وإلى الـ Memory مباشرة من غير ما الـ CPU يتدخل.

المشكلة بدون DMA: لو محتاج تنقل 1MB بيانات من كاميرا للذاكرة، الـ CPU هيفضل شغّال كله بالنقل ده — وهو شغّال بده مش هيقدر يعمل حاجة تانية.

الحل مع DMA: الـ DMA Controller بياخد مهمة النقل — الـ CPU بيقوله "انقل من X لـ Y" وبعدين الـ CPU يروح يعمل حاجات تانية. لما النقل يخلص الـ DMA بيعمل Interrupt للـ CPU يقوله "خلصت".

نقل بدون DMA

CPU بيقرأ من الـ Peripheral → بيكتب في الذاكرة → يكرر آلاف المرات.

الـ CPU مشغول 100% بالنقل — ما بيعملش أي حاجة تانية.

Slow + CPU Overhead عالي.

نقل مع DMA

CPU بيقول للـ DMA "انقل"، الـ DMA يشتغل، الـ CPU يعمل حاجات تانية.

لما DMA يخلص: Interrupt للـ CPU.

Fast + CPU Overhead قليل.

أمثلة استخدام DMA:

نقل Audio data من microphone للذاكرة. نقل Video frames من كاميرا للذاكرة. نقل البيانات من/إلى USB. نقل البيانات في الـ Network cards.

🔍 JTAG — Joint Test Action Group

JTAG هو standard interface بيستخدم في 3 أغراض رئيسية:

1. Debugging: بيخليك تتوقف عند نقطة معينة في الكود (Breakpoint) وتشوف قيم الـ Registers والـ Memory — من غير ما تحتاج تفك الـ Chip من الدائرة.

2. Programming: بتنزّل الكود على الـ Microcontroller من خلال JTAG — بديل لأساليب Programming التانية.

3. Testing / Boundary Scan: بتتحقق إن الـ connections على الـ PCB (اللحام) صح — مهم جداً في مرحلة التصنيع.

JTAG بيستخدم 4-5 pins: TDI (Test Data In)، TDO (Test Data Out)، TCK (Test Clock)، TMS (Test Mode Select)، TRST (optional Reset).

🧅 OS vs Kernel — الفرق

Kernel هو القلب أو النواة — الجزء الأساسي في الـ OS اللي بيشتغل مباشرة على الـ Hardware. ده اللي بيتحكم في كل حاجة على مستوى منخفض.

Operating System = Kernel + User Interface + Libraries + Applications. يعني الـ OS أكبر من الـ Kernel.

الـ Kernel بيدير: Memory Management, Process/Task Scheduling, Device Control, Interrupt Handling.

💡 تشبيه بسيط

⚡ Bare-Metal Programming

Bare-Metal يعني إنك بتكتب الكود مباشرة على الـ Hardware من غير OS. الكود بتاعك هو اللي بيتحكم في كل حاجة.

Arduino Uno في الغالب Bare-Metal — مش فيه OS، الكود بيشتغل مباشرة على الـ ATmega328.

مميزات: أسرع، أقل استهلاك للذاكرة، أبسط للأنظمة الصغيرة.

عيوب: أصعب في الـ Development، مش بيعمل Multitasking بشكل طبيعي.

Communication Protocols
كيف الأجهزة بتتكلم مع بعض — UART, SPI, I2C وأكتر

📡 مفاهيم أساسية قبل ما ندخل في البروتوكولات

Serial vs Parallel: Serial = بتبعت الـ bits واحدة ورا التانية على سلك واحد. Parallel = بتبعت كذا bit في نفس الوقت على أسلاك متعددة. الـ Serial أسهل ومقاوم أكثر للمسافات الطويلة.

Synchronous vs Asynchronous: Synchronous = فيه Clock مشترك بين الـ sender والـ receiver — بيتزامنوا عليه. Asynchronous = مفيش Clock مشترك — كل جهاز عنده clock خاص، ولازم يتفقوا على baud rate.

Half-Duplex vs Full-Duplex: Half = إرسال أو استقبال في وقت واحد، مش الاتنين. Full = إرسال واستقبال في نفس الوقت (زي التليفون).

Baud Rate: عدد الـ Symbols (تغييرات الإشارة) في الثانية. في أغلب الحالات = Bit Rate.

📌 من الامتحان الفعلي (سؤال 12)

"Which is a key advantage of serial communication over parallel?" — الإجابة: It is less prone to signal degradation and crosstalk over long distances.

سؤال 13: المشكلة الكبيرة في Parallel over long distances: Timing skew — bits arrive at slightly different times causing data corruption.

1️⃣ UART — Universal Asynchronous Receiver/Transmitter

تعريفه: بروتوكول تواصل تسلسلي Asynchronous — مفيش clock مشترك. كل جهاز بيستخدم clock خاص بيه ولازم يتفقوا على Baud Rate واحد.

الأسلاك: 2 أسلاك فقط: TX (Transmit) و RX (Receive).

Master/Slave: لأ — Peer-to-Peer، كل جهاز متساوي.

Duplex: Full-Duplex (بيرسل ويستقبل في نفس الوقت على سلكين منفصلين).

Data Frame: Start bit → 5-9 Data bits → Optional Parity bit → 1-2 Stop bits.

Error Detection: Parity bit (بيكتشف بعض الأخطاء) + Framing errors (لو stop bit مش في مكانه).

الـ Baud Rates الشائعة: 9600, 115200 bps.

مميزاته: بسيط جداً، 2 سلك بس، Full-Duplex.

عيوبه: ما بيدعمش Multi-slave، بطيء نسبياً، محتاج نفس الـ baud rate على الجهازين.

استخدامه: GPS modules, Bluetooth modules (HC-05), GSM modules, التواصل مع الكمبيوتر.

⚠️ TRAP سؤال 17 من الامتحان

"Which is true about UART?" — الإجابة الصح: UART supports full-duplex communication using separate TX and RX lines.

❌ "UART is synchronous" — غلط، هو asynchronous.

❌ "Both devices must operate at different baud rates" — غلط! لازم نفس الـ baud rate.

2️⃣ USART — Universal Synchronous/Asynchronous Receiver/Transmitter

USART = UART + القدرة على الشغل Synchronous أيضاً.

في الـ Synchronous mode، فيه Clock line إضافية. في الـ Asynchronous mode، بيشتغل زي UART عادي.

ATmega328 (Arduino Uno) فيه USART — يعني يقدر يشتغل بالاتنين.

⚠️ TRAP مهمة جداً

UART ≠ USART. الـ USART هو superset — بيعمل كل حاجة UART بيعملها + أكتر (Synchronous mode).

لو السؤال قال "UART or USART" — الـ USART هو الأشمل.

3️⃣ SPI — Serial Peripheral Interface

تعريفه: بروتوكول Synchronous (فيه Clock مشترك) بيستخدم architecture Master/Slave.

الأسلاك الأساسية الـ 4:

• MOSI (Master Out Slave In) — بيانات من Master للـ Slave.

• MISO (Master In Slave Out) — بيانات من Slave للـ Master.

• SCK/CLK (Serial Clock) — الـ Clock اللي الـ Master بيولده.

• SS/CS (Slave Select / Chip Select) — بيحدد أنهي Slave بيشتغل دلوقتي.

Master/Slave: نعم — Master واحد + عدة Slaves.

Duplex: Full-Duplex.

السرعة: الأسرع في المقارنة — ممكن يوصل لـ 80MHz على ESP8266.

مميزاته: سريع جداً، Full-Duplex.

عيوبه: محتاج سلك SS لكل Slave إضافي — كتير Slaves = كتير pins.

استخدامه: SD Card modules, TFT Displays, ADC chips, SPI Flash memories.

⚠️ TRAP من الامتحان (سؤال 21-23)

سؤال 21: "Signal lines in standard 4-wire SPI?" — الإجابة: MISO, MOSI, SCK, and CS (or SS).

سؤال 22: "Nature of SPI?" — الإجابة: SPI is synchronous, uses shared clock line (SCK) for synchronization.

سؤال 23: "Why is SPI synchronous even though it lacks start/stop bits?" — الإجابة: The master provides a shared clock signal (SCK).

4️⃣ I²C — Inter-Integrated Circuit

تعريفه: بروتوكول Synchronous بيستخدم سلكين بس ويدعم Multiple Masters و Multiple Slaves على نفس الـ Bus.

الأسلاك:

• SDA (Serial Data) — بيانات.

• SCL (Serial Clock) — الـ Clock.

كيف بيميّز بين الـ Devices؟ كل Device عنده 7-bit address فريد.

Master/Slave: نعم — يدعم Multiple Masters و Multiple Slaves.

Duplex: Half-Duplex (SDA مشترك للاتجاهين).

السرعة: أبطأ من SPI. Standard: 100kHz, Fast: 400kHz, High-Speed: 3.4MHz.

Arbitration: لما أكتر من Master بيحاولوا يبعتوا في نفس الوقت — I²C فيه آلية Non-Destructive Arbitration: المـ Master اللي بيكتب '1' وهو شايف '0' على الـ SDA يعرف إنه خسر الـ arbitration ويوقف.

Clock Stretching: الـ Slave يقدر يمسك الـ SCL على Low عشان يأخر الـ Master ويكسب وقت.

مميزاته: سلكين بس لأي عدد devices، multi-master.

عيوبه: أبطأ من SPI، Half-Duplex.

استخدامه: Temperature sensors, OLED Displays, RTC modules, IMU sensors.

⚠️ TRAP أسئلة 26-31 من الامتحان

سؤال 27: كل الإجابات A, B, C, D صح — الإجابة هي E (ALL).

سؤال 28: "What happens during I²C arbitration?" — الإجابة: The master that first releases the SDA line loses arbitration.

سؤال 30: "Which is NOT true?" — الإجابة: A: Arbitration happens only during the slave address transmission phase — غلط! Arbitration يحصل كمان خلال الـ Data Phase.

سؤال 31: "Incorrect about failed I²C arbitration?" — الإجابة: A: A master losing arbitration must immediately send a STOP — ده غلط! الـ Master الخاسر بيوقف التحويل بدون STOP.

5️⃣ TWI — Two Wire Interface

TWI هو نفس I²C تماماً. Atmel (صانع ATmega) سمّاه TWI عشان يتجنب مشاكل حقوق النشر مع Philips (اللي اخترعت I²C).

لما الدكتور يسأل عن TWI — إجاباتك عن I²C صح تماماً.

6️⃣ LIN — Local Interconnect Network

تعريفه: بروتوكول تواصل تسلسلي Asynchronous بسيط وأرخص من CAN.

الأسلاك: سلك واحد بس (Single wire) + Ground.

Master/Slave: نعم — Master واحد + حتى 16 Slave.

Duplex: Half-Duplex.

السرعة: بطيء — حتى 20 kbps.

استخدامه: داخل السيارات — للتحكم في الإضاءة، المرايا، مقاعد، مكيفات.

أرخص من CAN ومناسب للمهام البسيطة داخل السيارة.

7️⃣ OWI — One Wire Interface

تعريفه: بروتوكول Asynchronous بيستخدم سلك واحد فقط للبيانات (بالإضافة لـ Ground).

Master/Slave: نعم.

السرعة: بطيء جداً (حوالي 16.3 kbps).

استخدامه: Temperature sensors (DS18B20)، الأجهزة اللي محتاجة أقل pins ممكنة.

ممكن يوصل الجهاز من الـ data wire نفسها (Parasitic Power).

8️⃣ LonWorks — Local Operating Network

تعريفه: بروتوكول للـ Industrial Automation والـ Building Control.

بيستخدم في التحكم في المباني (إضاءة، تكييف، كهرباء)، المصانع، الـ Smart Grid.

يدعم عدد ضخم من الـ Nodes على نفس الـ Network.

أبطأ وأعقد من I²C و SPI لكن مصمم للـ Industrial Scale.

FPGA vs ASIC vs DSP vs GPP
أنواع المعالجات والحلول الإلكترونية

💡 تعريف سريع للأربعة

GPP (General Purpose Processor): المعالج العام — زي Intel Core أو ATmega. مرن جداً، تبرمجه لأي مهمة. لكن مش متخصص.

ASIC (Application-Specific Integrated Circuit): دائرة متكاملة مصممة لمهمة واحدة محددة فقط. تُصمم من الصفر، بعد التصنيع مش ممكن تغيّرها.

FPGA (Field-Programmable Gate Array): دائرة بتقدري تبرمجيها بعد التصنيع عدد غير محدود من المرات. فيها Logic Blocks قابلة للتهيئة.

DSP (Digital Signal Processor): معالج متخصص في معالجة الإشارات الرقمية — فيه وحدات خاصة للعمليات الرياضية المتكررة (زي Multiply-Accumulate).

🎯 DSP Features الخاصة

Dedicated Multiplier/MAC (Multiply-Accumulate): وحدة hardware مخصصة للضرب والجمع المتكرر — مهمة جداً في الـ filtering.

Saturation Arithmetic: لما النتيجة تعدي الحد الأقصى، بدل ما تـ overflow تفضل عند الحد الأقصى — مهم في الـ audio processing.

Pipelining: تنفيذ أكتر من تعليمة في نفس الوقت على مراحل.

SIMD (Single Instruction Multiple Data): تعليمة واحدة بتشتغل على بيانات كتير في نفس الوقت — مثالية للـ Signal Processing.

VLIW (Very Long Instruction Word): كل "تعليمة" بتحتوي على أكتر من عملية تتنفذ في نفس الـ clock.

Von Neumann: ده مش feature في DSP — DSP بيستخدم Harvard Architecture (ذاكرتين منفصلتين للكود والبيانات). من الامتحان: "Which is NOT a typical DSP feature? Von Neumann memory architecture".

void setup() { // بتشتغل مرة واحدة بس عند بداية التشغيل أو الـ Reset // بتستخدم للـ initialization: pinMode(LED_PIN, OUTPUT); Serial.begin(9600); } void loop() { // بتشتغل للأبد — بتكرر للنهاية // ده جسم البرنامج الرئيسي digitalWrite(LED_PIN, HIGH); delay(1000); }

setup() = run once at power-up or reset. loop() = runs continuously forever.

🔘 Pull-Up vs Pull-Down Resistors

المشكلة: لما Button مش مضغوط، الـ pin "طائر" (floating) — بيقرأ قيم عشوائية.

الحل: مقاومة بتثبّت الـ pin على قيمة معروفة.

Pull-Down Resistor

الـ pin متوصل بـ مقاومة للـ GND.

Button مش مضغوط → Pin = LOW (0V).

Button مضغوط → Pin = HIGH (VCC).

كود: pinMode(pin, INPUT) + مقاومة خارجية.

بتقرأ: HIGH لما مضغوط.

Pull-Up Resistor (أو INPUT_PULLUP)

الـ pin متوصل بـ مقاومة للـ VCC.

Button مش مضغوط → Pin = HIGH (VCC).

Button مضغوط → Pin = LOW (GND).

كود: pinMode(pin, INPUT_PULLUP).

بتقرأ: LOW لما مضغوط (عكسي!).

⚠️ TRAP مهمة جداً (من الامتحان الفعلي)

Button 1: Pull-DOWN → reads HIGH when pressed.

Button 2: Pull-UP (internal) → reads LOW when pressed.

عشان تكتشف Button 2 مضغوط: if (!digitalRead(BUTTON2_PIN)) — لاحظي علامة النفي (!).

سؤال 4: الإجابة B: if (!digitalRead(BUTTON2_PIN)).

⚡ Debouncing — إزالة اهتزاز الزر

لما بتضغطي Button، مش بيدي إشارة نظيفة — بيكون فيه "bounce" (اهتزازات سريعة) لمدة ~5-20ms.

من غير debouncing، الـ Arduino ممكن يشوف ضغطة واحدة كـ 5-10 ضغطات.

طريقة الـ while loop للـ debouncing (من الامتحان):

// Button 1: Pull-DOWN → HIGH when pressed if (digitalRead(BUTTON1_PIN)) { counter = (counter + 1) % maxStates; while (digitalRead(BUTTON1_PIN)); // انتظر لحد ما الزر يتحرر } // Button 2: Pull-UP → LOW when pressed if (!digitalRead(BUTTON2_PIN)) { counter = (counter - 1 + maxStates) % maxStates; while (!digitalRead(BUTTON2_PIN)); // انتظر لحد ما الزر يتحرر }
⚠️ TRAP أسئلة 3 و 5 من الامتحان

سؤال 3: "Debounce Button 1 (pull-down) after its block?" — الإجابة: C) while (digitalRead(BUTTON1_PIN)); — تنتظر لحد ما يبقى LOW (متحرر).

سؤال 5: "Debounce Button 2 (pull-up) after its block?" — الإجابة: A) while (!digitalRead(BUTTON2_PIN)); — تنتظر لحد ما يبقى HIGH (متحرر).

🔢 أرقام Arduino Uno

  • MCU: ATmega328P
  • Clock: 16 MHz
  • Flash: 32 KB
  • SRAM: 2 KB
  • EEPROM: 1 KB
  • Digital I/O: 14 pins
  • Analog In: 6 (A0-A5)
  • ADC: 10-bit
  • Operating V: 5V
  • Max pin current: 40 mA
  • Language: C++

🔢 أرقام ESP8266 NodeMCU

  • CPU: Xtensa LX106 (32-bit)
  • Clock: 80-160 MHz
  • RAM: 80 KB
  • Flash: 4 MB
  • ADC: 1 channel فقط (A0)
  • ADC: 10-bit
  • Operating V: 3.3V
  • Max pin current: 12 mA
  • Wi-Fi: 802.11 b/g/n
  • GPIO6-11: محجوزة للـ Flash
  • SPI clock: 40 MHz
  • I²C clock: 1 MHz

📡 Communication Protocols

  • UART: Async, Full-Duplex, 2 wires, P2P
  • USART: Async+Sync, Full-Duplex
  • SPI: Sync, Full-Duplex, 4 wires, M/S
  • I²C/TWI: Sync, Half-Duplex, 2 wires, Multi
  • LIN: Async, Half-Duplex, 1 wire, 16 slaves
  • OWI: Async, 1 wire
  • SPI الأسرع: MBps
  • I²C Standard: 100kHz, Fast: 400kHz
  • I²C arbitration = Non-destructive
  • TWI = I²C (نفس الحاجة)

🏗️ ASIC vs FPGA vs GPP vs DSP

  • Lowest power → ASIC
  • Highest power → GPP
  • Lowest unit price → ASIC
  • Highest price → FPGA
  • Easiest implementation → GPP
  • Hardest implementation → ASIC
  • Fastest time to market → GPP
  • Slowest time to market → ASIC
  • Reprogrammable → FPGA
  • Signal processing → DSP
  • NOT DSP feature → Von Neumann

⏱️ Real-Time Systems

  • Hard RT: Deadline strict → Catastrophic if missed
  • Soft RT: Deadline important → Performance degrades
  • Hard RT examples: Airbag, Pacemaker, Flight control
  • Soft RT examples: Video streaming, Email
  • Real-Time ≠ Fast (= Predictable)
  • RTOS that guarantees → Hard Real-Time RTOS
  • Deadline = time by which task must complete

💾 Memory

  • Volatile: RAM (SRAM, DRAM)
  • Non-Volatile: ROM, Flash, EEPROM
  • SRAM: Faster, Flip-Flops, no refresh
  • DRAM: Cheaper, Capacitors, needs refresh
  • ROM: Programmed at manufacture
  • PROM: One-Time Programmable
  • EPROM: Erase with UV Light
  • EEPROM: Erase electrically (Byte)
  • Flash: Erase electrically (Block) → faster
  • ATmega: Flash=Code, SRAM=Variables, EEPROM=Settings

🔘 Button Logic

  • Pull-DOWN → HIGH when pressed
  • Pull-UP → LOW when pressed (عكسي!)
  • INPUT_PULLUP = internal pull-up
  • Detect Pull-UP: if(!digitalRead(pin))
  • Debounce Button 1 (pull-down): while(digitalRead(pin));
  • Debounce Button 2 (pull-up): while(!digitalRead(pin));
  • t_delay = 1000/f (Hz to ms)
  • Gray Code: 1 bit changes at a time
  • Gray = counter ^ (counter >> 1)

🧠 CPU Components

  • CPU = Control Unit + ALU
  • ALU = Arithmetic + Logic operations
  • CU = Controls instruction sequence
  • PC = Next instruction address
  • IR = Current instruction
  • Zero Flag: result = 0
  • Carry Flag: overflow/borrow
  • Cycle: Fetch → Decode → Execute
  • Brain of computer = CPU
  • DMA = Direct Memory Access (no CPU)
  • JTAG = Debug + Program + Test
